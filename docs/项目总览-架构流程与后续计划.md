# 项目总览：架构流程与后续计划

本文是项目级入口，承载项目定位、架构、当前成果、核心流程、工程判断和后续 Todo。具体接口、运行命令、配置项、协议字段和排障细节请继续阅读对应专题文档。

## 项目定位

QA Test Framework 是一套面向 Unity 应用的运行时自动化测试调用框架。Unity 运行时通过 `[QaTest]` 暴露可测试能力，register server 负责注册、调度、状态维护和结果广播，Web 控制台服务人工操作，qamcp 服务 AI Agent 自动调用。

项目目标是把 Unity 运行时测试能力标准化、服务化，让人工 QA、客户端开发和 AI Agent 都能通过统一链路发现实例、执行方法、编排请求序列和查看结果。

## 当前成果

- Unity 客户端可在 QA 功能启用后自动连接服务端，扫描并注册带 `[QaTest]` 标记的方法；Editor 默认启用，Player 默认关闭。
- Rust register server 是唯一主实现，负责客户端注册、状态维护、请求调度、控制锁、执行超时、执行历史和结果广播。
- Web 控制台支持查看在线 Unity 实例、执行单个方法、编排请求序列和查看执行记录。
- qamcp 将 QA 能力开放给 AI Agent，支持实例发现、方法搜索、单步执行、序列执行、事件监听和结果查询。
- WebSocket 状态收敛已覆盖业务 heartbeat、执行超时、控制锁、late result、主动停止、本地 busy 和只读查询类并行执行。
- 可选 MariaDB/MySQL 执行历史旁路存档已落地，用于长期查询和审计；默认仍使用内存最近历史。

### Unity Client

`unityclient` 使用 `QaTestBootstrap` 在 QA 功能启用时自动创建 `QaTestClient`，通过 WebSocket 连接 register server，并扫描当前 AppDomain 中带 `[QaTest]` 的方法。

核心职责：

- 生成并持久化 `clientId` 和 `clientName`。
- 提供 QA 功能启用开关；Editor 默认启用，Player 默认关闭。
- 解析命令行、PlayerPrefs 或 Inspector 中配置的 server WebSocket 地址。
- 发送 `register` 和 `heartbeat`，同步客户端信息、方法元数据和本地 busy 状态。
- 接收 `execute` 指令，在 Unity 主线程执行目标方法并回传 `qa_result`。
- 支持普通返回值、`Task`、`Task<T>`、`IEnumerator`、`QaTestCoroutineReturn` 和 `QaTestCoroutineResult`。
- 支持把 `failed:<reason>`、`false`、`ok=false` 或失败态 `status` 的结构化结果映射为业务失败。
- 默认保持单客户端串行执行；标记 `AllowParallelExecution` 的只读查询方法可并行执行。

### Register Server

`registerserver/rustserver` 是注册调度中心的唯一主实现。`registerserver/server` 是冻结的 Node legacy，仅作为历史行为参考和紧急本地回退入口，不再同步新能力、协议字段或常规修复。

核心职责：

- 维护 Unity 长连接、在线状态、业务可用状态、本地 busy 状态和注册方法列表。
- 维护 Web/MCP 控制端连接和控制端身份。
- 使用控制锁避免多个控制端同时操作同一 Unity 实例。
- 接收 Web/MCP 的执行请求并转发给目标 Unity。
- 为单次执行和序列步骤创建 pending 状态并做超时收敛。
- 维护最近 200 条内存执行历史，并支持可选 MariaDB/MySQL 旁路存档。
- 向 Web/MCP 广播注册、断开、锁状态、执行状态、序列状态和结果事件。
- 生产环境下托管 `QA_CLIENT_DIST_DIR` 指向的 Web 控制台构建产物。

### Web 控制台

`registerserver/client` 是人工操作控制台。它通过 HTTP 获取快照，通过 WebSocket 接收实时事件并提交执行请求。

核心职责：

- 展示在线 Unity 实例、业务可用状态、运行状态、控制锁状态和注册方法列表。
- 执行单个 `[QaTest]` 方法。
- 编排并顺序执行请求序列。
- 停止运行中的请求或请求序列。
- 展示最近执行历史和序列执行结果。
- 使用 sessionStorage 维护当前 Web 控制端 ID。

### qamcp

`qamcp` 是面向 AI Agent 的 stdio MCP server。它不直接连接 Unity，而是复用 register server 的 HTTP 和 WebSocket 能力。

核心职责：

- 将 QA 能力包装为 MCP tools。
- 读操作通过 HTTP API 完成，例如健康检查、客户端列表、方法搜索和历史查询。
- 执行、停止、等待和事件监听通过 `WS /ws?role=web` 完成。
- 使用 `qamcp config set serverUrl <url>` 指向目标 register server，默认 `http://localhost:3000`。
- 使用 `QA_CONTROLLER_ID` 控制 MCP 控制端身份，未设置时自动生成。
- 支持临时事件监听和可轮询的长连接事件会话。

## 交互架构

```text
AI Agent
  |
  | stdio MCP tools
  v
qamcp
  |
  | HTTP: /api/health, /api/unity-clients, /api/results
  | WS:   /ws?role=web&controllerType=mcp&controllerId=...
  v
registerserver/rustserver <------------ registerserver/client
  ^                                             |
  | WS: /ws?role=unity                          | HTTP + WS
  |
unityclient / QaTestClient
```

Web 控制台和 MCP 工具在 server 看来都属于 `role=web` 控制端。差异主要在 `controllerType`：Web 控制台是 `web`，qamcp 是 `mcp`。

内网部署下默认使用 `QA_ACCESS_SCOPE=private` 限制来源网络，并可用 `QA_WEB_CONSOLE_TOKEN` 给浏览器控制台增加轻量登录。该 token 只影响浏览器控制台登录和 WebSocket 握手；Unity 和 qamcp 保持免 token 调用。

## 核心流程

### 启动和注册

1. 启动 `registerserver/rustserver`，默认监听 `0.0.0.0:3000` 并限制为内网来源。
2. Unity 进入 Play Mode 或 Player 启动；QA 功能启用时，`QaTestBootstrap` 自动创建 `QaTestClient`。
3. `QaTestClient` 解析连接地址，默认连接 `ws://localhost:3000/ws?role=unity`。
4. Unity 刷新 `QaTestRegistry` 并扫描当前 AppDomain。
5. Unity 发送 `register`，包含 `clientId`、名称、IP、平台、Unity 版本、busy 状态和方法列表。
6. server 保存或替换同 `clientId` 的 Unity 连接，返回 `registered` 并广播 `unity_registered`。
7. Unity 后续持续发送 `heartbeat`，server 更新 `lastSeenAt` 和本地 busy 状态；busy 变化时广播 `unity_state_changed`。

### 单次执行

1. Web 控制台或 qamcp 连接 `WS /ws?role=web`。
2. 控制端发送 `execute`，指定 `clientId`、`methodId`、`methodName`、`arguments`，可选 `timeoutMs` 和 `allowParallelExecution`。
3. server 校验 Unity 是否在线、业务可用、未被其他控制端锁定且没有冲突执行。
4. server 获取控制锁，生成 `requestId`，写入 running 历史并广播 `execution_started`。
5. server 向 Unity 下发 `execute`。
6. Unity 切回主线程执行目标方法并发送 `qa_result`。
7. server 用同一 `requestId` 覆盖 running 历史，广播结果，并在无其他运行任务时释放控制锁。

### 请求序列

1. Web 控制台或 qamcp 通过 `execute_sequence` 提交 `sequenceId`、目标 `clientId`、`steps`、`stopOnFailure` 和 `stepDelayMs`。
2. server 校验目标 Unity 并获取控制锁。
3. server 对每个 step 生成独立 `requestId`，按顺序下发 `execute`。
4. 每步完成后广播 `qa_result` 和 `sequence_step_result`。
5. 如果 `stopOnFailure=true` 且步骤失败，server 停止后续步骤。
6. 序列完成、失败或取消后广播 `sequence_finished` 并释放控制锁。

## 状态和故障收敛

server 将 Unity 状态拆成几类：

- `online`: WebSocket 协议层连接是否健康。
- `available`: Unity 业务 heartbeat 是否正常。
- `locked`: 是否被某个 Web/MCP 控制端占用。
- `running`: 是否存在 server 已下发但未完成的执行。
- `clientBusy`: Unity 客户端本地是否正在执行请求。
- `allowParallelExecution`: 方法或请求声明允许并行执行时，用于只读查询类方法绕过普通单客户端互斥。

关键策略：

- WebSocket ping/pong 失败、close 或 error 会让 Unity 离线，并广播 `unity_disconnected`。
- Unity 业务 heartbeat 超时只标记 `available=false`，广播 `unity_unavailable`，不会立即删除连接。
- 后续收到 heartbeat、register 或其他 Unity 业务消息会恢复 `available=true`。
- 单次执行和序列步骤默认使用 `EXECUTION_TIMEOUT_MS` 超时；HTTP/WS 单次执行、请求序列顶层和单个 step 都可以传 `timeoutMs` 覆盖。
- 超时后 server 生成 failed 结果，写入历史并广播 `qa_result`。
- 超时后迟到的同 `requestId` 结果不会覆盖最终状态，会以 `qa_result_late` 形式广播。
- 控制端断线不取消已提交执行；锁会保留到执行结束或超时。
- 主动停止会把 server 侧 pending 执行或序列标记为 `cancelled`，但当前 Unity 端没有强制中断正在运行方法的机制。

## 当前工程判断

项目已经形成可用闭环，更接近内部工具和团队 PoC 到稳定内用之间的阶段。核心链路可跑通，主要工程风险集中在安全边界、测试覆盖、协议治理、停止语义和长期维护。

已处理的重要治理项：

- Unity Player 默认关闭 QA 功能，避免正式包默认暴露 QA WebSocket 能力。
- Web 控制台支持 `QA_WEB_CONSOLE_TOKEN` 轻量登录。
- Rust server 支持 `QA_LISTEN_HOST` 和 `QA_ACCESS_SCOPE`，默认限制为内网来源。
- Node server 已冻结为 legacy，不再作为双主实现维护。
- qamcp 已迁移为 Rust 实现，并支持 Windows exe 交付。
- Web 控制台已完成路由懒加载、Element Plus 按需导入和分包优化。
- 根目录按单一 Git 仓库管理，各模块源码、项目文档、版本矩阵和发布检查清单统一提交。
- 文档已拆分为项目总览、模块说明、配置、运行、能力边界、发布和存档专题。
