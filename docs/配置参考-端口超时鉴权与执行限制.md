# 配置参考：端口、超时、鉴权与执行限制

本文集中记录 QA Test Framework 当前可配置项。除特别说明外，命令都以仓库根目录 `c:\project\qatestframework` 为参照。

## 配置总览

### Register Server

`registerserver/rustserver` 是主服务端，配置通过环境变量读取，启动前设置即可。本地开发也可以复制 `registerserver/rustserver/.env.example` 为 `registerserver/rustserver/.env`，服务端启动时会自动加载该文件。服务端会先尝试读取当前工作目录下的 `.env`；如果没有找到，会再尝试读取 exe 同目录下的 `.env`，便于线上 Windows 拷贝部署。

| 配置项 | 默认值 | 说明 |
| --- | --- | --- |
| `PORT` | `3000` | Rust server HTTP 和 WebSocket 端口。 |
| `QA_LISTEN_HOST` | `0.0.0.0` | Rust server HTTP 和 WebSocket 监听地址。默认监听所有网卡，便于同一内网内的 Unity、Web 控制台和 qamcp 连接。 |
| `QA_ACCESS_SCOPE` | `private` | 访问范围。`private` 只允许本机、RFC1918 私网、链路本地和 IPv6 内网来源；`unrestricted` 不限制来源。 |
| `EXECUTION_TIMEOUT_MS` | `20000` | 单次执行和请求序列单步默认超时，单位毫秒。 |
| `UNITY_HEARTBEAT_STALE_MS` | `45000` | Unity 业务 heartbeat 超过该时间未更新后标记为不可用，单位毫秒。 |
| `WS_HEARTBEAT_INTERVAL_MS` | `15000` | 服务端 WebSocket ping/pong 检查间隔，单位毫秒。 |
| `QA_WEB_CONSOLE_TOKEN` | 未设置 | 可选 Web 控制台登录 token；未设置时控制台免登录。 |
| `QA_CLIENT_DIST_DIR` | 自动解析 | Web 控制台构建产物目录。未设置时优先使用 exe 同目录下的 `client/dist`，本地开发回退到仓库中的 `registerserver/client/dist`。 |
| `QA_ARTIFACT_DIR` | 当前工作目录下的 `artifacts` | 截图/附件 artifact 存储目录。目录不存在时 server 会自动创建。 |
| `QA_ARTIFACT_MAX_BYTES` | `20971520` | 单个 artifact 最大上传字节数，默认 20MB；超出返回 `413 Payload Too Large`。 |
| `QA_LOG_DIR` | 未设置 | 可选日志目录；设置后 Rust server 会额外写入按天切分的 JSON line 日志文件。 |
| `QA_LOG_PREFIX` | `registerserver` | 日志文件名前缀，仅在设置 `QA_LOG_DIR` 时生效。 |
| `QA_EXECUTION_ARCHIVE_MYSQL_ENABLED` | `false` | 是否启用 MariaDB/MySQL 执行历史旁路存档。 |
| `QA_EXECUTION_ARCHIVE_MYSQL_URL` | 未设置 | MariaDB/MySQL 连接串，只有启用存档时才需要。 |
| `QA_EXECUTION_ARCHIVE_MYSQL_MAX_CONNECTIONS` | `5` | 存档连接池最大连接数。 |
| `QA_EXECUTION_ARCHIVE_QUEUE_SIZE` | `10000` | 异步存档队列长度；队列满时丢弃存档任务并记录 warn。 |

启动示例：

```powershell
cd registerserver/rustserver
Copy-Item .env.example .env
# 按需编辑 .env
cargo run
```

也可以直接在当前终端设置环境变量：

```powershell
cd registerserver/rustserver
$env:PORT=3000
$env:QA_LISTEN_HOST="0.0.0.0"
$env:QA_ACCESS_SCOPE="private"
$env:EXECUTION_TIMEOUT_MS=20000
$env:UNITY_HEARTBEAT_STALE_MS=45000
$env:WS_HEARTBEAT_INTERVAL_MS=15000
$env:QA_WEB_CONSOLE_TOKEN="replace-with-internal-token"
$env:QA_CLIENT_DIST_DIR="C:\deploy\qa-register\client\dist"
$env:QA_ARTIFACT_DIR="C:\deploy\qa-register\artifacts"
$env:QA_ARTIFACT_MAX_BYTES=20971520
$env:QA_LOG_DIR="logs"
$env:QA_LOG_PREFIX="registerserver"
cargo run
```

真实 `.env` 文件不会提交到 git；需要共享配置项说明时只更新 `.env.example` 和本文档。

默认配置适合内网测试：服务端绑定 `0.0.0.0:<PORT>`，但 `QA_ACCESS_SCOPE=private` 会拒绝公网来源，只允许本机、私网、链路本地和 IPv6 内网来源访问 HTTP API 与 WebSocket。

如果只允许本机访问，可以设置：

```powershell
$env:QA_LISTEN_HOST="127.0.0.1"
```

如果确实需要放开来源限制，必须显式设置：

```powershell
$env:QA_ACCESS_SCOPE="unrestricted"
```

只有在外部网关、防火墙或反向代理已经完成访问控制时才建议使用 `unrestricted`。

Web 控制台静态文件目录通过 `QA_CLIENT_DIST_DIR` 配置。线上 Windows 不需要安装 Rust，可以把 `qa-register-rustserver.exe`、`.env` 和 `client/dist` 放在同一个部署目录：

```text
qa-register/
  qa-register-rustserver.exe
  .env
  client/
    dist/
      index.html
      assets/
```

这种结构下可以不设置 `QA_CLIENT_DIST_DIR`。如果前端文件放在其他目录，则在 `.env` 中写绝对路径。

截图和附件 artifact 默认写入服务进程当前工作目录下的 `artifacts` 目录。线上如果希望与程序目录、日志目录或数据盘保持固定位置，设置 `QA_ARTIFACT_DIR` 为绝对路径。server 会为每个 artifact 创建独立 UUID 子目录，并保存文件与 `metadata.json`。

### Web 控制台

生产环境由 Rust server 托管 `QA_CLIENT_DIST_DIR` 指向的 Web 控制台构建产物，访问地址跟随 `PORT`：

```text
http://localhost:3000/
```

开发环境使用 Vite：

| 配置项 | 默认值 | 说明 |
| --- | --- | --- |
| Vite dev server | `5173` | `registerserver/client/vite.config.js` 中固定为 `5173`。 |
| `/api` 代理 | `http://localhost:3000` | 开发环境代理到 Rust server。 |
| `/ws` 代理 | `ws://localhost:3000` | 开发环境代理到 Rust server WebSocket。 |

如果配置了 `QA_WEB_CONSOLE_TOKEN`，浏览器打开控制台时会先显示登录页。登录成功后服务端写入 HttpOnly cookie，并在浏览器控制台 WebSocket 握手时校验该 cookie。

该 token 只保护浏览器控制台。Unity 客户端和 qamcp 在当前内网部署模型下不需要额外传 token。

### Unity Client

Unity 运行时客户端配置来源包括 Inspector、命令行、环境变量和 PlayerPrefs。

| 配置项 | 默认值 | 来源 | 说明 |
| --- | --- | --- | --- |
| `enableInEditor` | `true` | Inspector | Editor 下默认启用 QA 客户端。 |
| `enableInPlayer` | `false` | Inspector | Player 下默认关闭 QA 客户端。 |
| `serverIP` / `serverPort` | `localhost` / `3000` | Inspector / PlayerPrefs / API / 命令行 URL | Unity WebSocket 服务地址；命令行 `--qa-server-url` 未带 `role` 时客户端会补 `role=unity`。Player 下推荐用 `QaTestClient.SetIpAndPort(...)` 写入 PlayerPrefs。 |
| `clientName` | `clientId` 前 8 位 | Editor: Inspector / `qatest.config.txt` / API；Player: PlayerPrefs / API | Editor 下如果项目根目录没有 `qatest.config.txt`，继续用 `clientId` 推导默认名且不自动写本地文件；Inspector 修改后会创建或更新 `qatest.config.txt`，后续优先读取其中的 `clientName` 并同步到 Inspector。Player 下外部配置只认 PlayerPrefs `QaTest.ClientName`。 |
| `reconnectDelaySeconds` | `2` | Inspector | 连接失败后的重连间隔。 |
| `heartbeatSeconds` | `10` | Inspector | Unity 业务 heartbeat 发送间隔。 |
| `connectTimeoutSeconds` | `10` | Inspector | Unity WebSocket 连接超时。 |

Player 中需要显式启用 QA 功能：

```powershell
Game.exe --qa-test-enable --qa-server-url ws://server-host:3000/ws
```

可用配置方式：

| 目标 | 配置方式 |
| --- | --- |
| 启用 QA | `--qa-test-enable`、`--qa-test-enabled=true`、`--qa-enable`、`--qa-enabled=true`、环境变量 `QA_TEST_ENABLED=1`、PlayerPrefs `QaTest.Enabled=1`、运行时 API `QaTestClient.SetGlobalEnabled(true)` |
| 关闭 QA | `--qa-test-disable`、`--qa-test-enabled=false`、`--qa-disable`、`--qa-enabled=false`、环境变量 `QA_TEST_ENABLED=0`、PlayerPrefs `QaTest.Enabled=0`、运行时 API `QaTestClient.SetGlobalEnabled(false)` |
| 覆盖服务地址 | `--qa-server-url ws://host:3000/ws` 或 `QaTestClient.SetIpAndPort(host, 3000)` |
| 固定客户端名称 | Editor: `QaTestClientName.Set(..., persist: true)`、内置 QA 方法 `设置客户端名称` 或修改 `qatest.config.txt`；Player: `QaTestClient.SetClientName(...)` 或内置 QA 方法 |
| Player 启动自动连接 | `QaTestClient.SetAutoConnectOnStartup(true)` 写入 PlayerPrefs `QaTest.AutoConnectOnStartup=1` |

### qamcp

`qamcp` 通过同目录的 `qamcp.config.json` 连接 register server。配置文件位于当前运行的 `qamcp.exe` 同目录；本地 `cargo run` 调试时可用 `QAMCP_CONFIG_DIR` 临时指定配置目录。

| 配置项 | 默认值 | 说明 |
| --- | --- | --- |
| `serverUrl` | `http://localhost:3000` | register server 地址。通过 `qamcp config set serverUrl <url>` 写入 `qamcp.config.json`，可传 `http/https/ws/wss`。 |
| `QA_CONTROLLER_ID` | `qamcp:<uuid>` | MCP 控制端身份。需要跨进程复用同一控制锁身份时设置。 |

示例：

```powershell
qamcp config set serverUrl http://localhost:3000
$env:QA_CONTROLLER_ID="my-agent-session"
```

qamcp 连接不受 `QA_WEB_CONSOLE_TOKEN` 影响。

## 日志配置

Rust server 默认向 stdout 输出 JSON line 结构化日志，不需要额外配置。设置 `QA_LOG_DIR` 后，server 会继续输出 stdout，同时用 `tracing-appender` 写入按天切分的日志文件。

```powershell
$env:QA_LOG_DIR="logs"
$env:QA_LOG_PREFIX="registerserver"
cargo run
```

日志文件位于 `QA_LOG_DIR` 目录下，文件名形如 `<QA_LOG_PREFIX>.YYYY-MM-DD`，例如 `registerserver.2026-05-08`。当前只负责按天切分，不自动清理旧日志。每条日志仍是一行 JSON，格式和 stdout 保持一致。

每条日志至少包含：

- `time`: ISO 8601 UTC 时间。
- `level`: `info`、`warn` 或 `error`。
- `event`: 事件名。

执行相关日志会尽量携带以下关联字段，便于排查单次执行、请求序列、控制锁和迟到结果：

- `requestId`
- `clientId`
- `controllerId`
- `controllerType`
- `sequenceId`
- `methodId`

示例：

```json
{"time":"2026-05-08T10:00:00.000Z","level":"info","event":"execution_started","requestId":"uuid","clientId":"client-1","controllerId":"web:abc","controllerType":"web","sequenceId":"seq-1","methodId":"Ping()","timeoutMs":20000}
```

当前已覆盖的主要事件包括：

- server 生命周期：`server_started`、`server_failed`、`shutdown_signal_received`。
- 连接状态：`unity_registered`、`unity_disconnected`、`unity_available`、`unity_unavailable`、`unity_state_changed`、`controller_connected`、`controller_disconnected`。
- 单次执行和锁：`execution_started`、`execution_finished`、`execution_rejected`、`execution_send_failed`、`execution_late_result`、`execution_unexpected_result`、`client_locked`、`client_unlocked`。
- 请求序列：`sequence_started`、`sequence_step_started`、`sequence_step_finished`、`sequence_step_rejected`、`sequence_finished`、`sequence_rejected`。
- 停止操作：`stop_execution_requested`、`stop_sequence_requested`、`stop_rejected`。
- 执行历史存档：`execution_archive_enabled`、`execution_archive_disabled`、`execution_archive_enqueue_failed`、`execution_archive_write_failed`。

## 超时配置

### 服务端执行超时

`EXECUTION_TIMEOUT_MS` 是服务端默认执行超时：

- HTTP `POST /api/unity-clients/:clientId/execute` 未传 `timeoutMs` 时使用该值。
- WebSocket `execute` 未传 `timeoutMs` 时使用该值。
- WebSocket `execute_sequence` 的每个 step 未传 `timeoutMs` 时使用该值。
- 请求序列顶层 `timeoutMs` 可作为所有 step 的默认超时；单个 step 的 `timeoutMs` 优先级更高。

超时后 server 会生成 failed `qa_result`，释放对应控制锁。Unity 后续迟到的同 `requestId` 结果不会覆盖最终状态，会广播为 `qa_result_late`。

### qamcp 工具等待超时

qamcp 的工具层等待超时独立于 server 执行超时：

| 工具场景 | 默认值 | 可配置方式 |
| --- | --- | --- |
| HTTP 请求 register server | `10000` ms | 当前为工具内常量。 |
| WebSocket 连接和停止确认 | `10000` ms | 停止类 tool 可传 `timeoutMs`，范围 `1000` 到 `30000`。 |
| 单次执行等待结果 | `70000` ms | `qa_execute_method.timeoutMs`，范围 `1000` 到 `300000`。 |
| 请求序列等待完成 | `180000` ms | `qa_execute_sequence.timeoutMs`，范围 `1000` 到 `600000`。 |
| 等待结果 | `180000` ms | `qa_wait_for_result.timeoutMs`，范围 `1000` 到 `600000`。 |
| 临时事件监听 | `10000` ms | `qa_watch_events.durationMs`。 |
| 长连接事件会话 TTL | `600000` ms | `qa_open_event_session.ttlMs`。 |

工具层超时只影响 qamcp 等待返回的时间；server 侧已经下发的执行仍按 server 执行超时或 Unity 回包收敛。

## Heartbeat 配置

当前有两类 heartbeat：

- WebSocket 协议层 ping/pong：由 server 按 `WS_HEARTBEAT_INTERVAL_MS` 主动检查 Unity 和 Web/MCP socket 是否仍在线。
- Unity 业务 heartbeat：由 Unity `QaTestClient` 按 `heartbeatSeconds` 发送，表示 Unity 主循环和 QA 客户端业务逻辑仍在工作。

推荐关系：

```text
Unity heartbeatSeconds < UNITY_HEARTBEAT_STALE_MS
WS_HEARTBEAT_INTERVAL_MS < UNITY_HEARTBEAT_STALE_MS
```

当前默认值为：

```text
heartbeatSeconds = 10 秒
WS_HEARTBEAT_INTERVAL_MS = 15 秒
UNITY_HEARTBEAT_STALE_MS = 45 秒
```

Unity 业务 heartbeat 超时后，server 只把客户端标记为 `available=false` 并广播 `unity_unavailable`，不会立即删除连接。后续收到 `heartbeat`、`register` 或其他 Unity 业务消息后会恢复为 `available=true`。

## Token 和安全边界

当前 token 只用于 Web 控制台轻量登录：

```powershell
$env:QA_WEB_CONSOLE_TOKEN="replace-with-internal-token"
```

- 未设置时，Web 控制台免登录。
- 设置后，浏览器控制台需要输入 token。
- 登录后服务端写入 HttpOnly cookie。
- 浏览器控制台 WebSocket 握手会校验 cookie。
- Unity WebSocket 和 qamcp 不校验该 token。

当前项目按内网可信部署设计。默认 `QA_ACCESS_SCOPE=private` 会限制来源网络，但同一内网内任意能访问 server 端口的人仍可直接调用 HTTP API、伪造 Unity WebSocket 或使用 qamcp 协议。若后续开放到非可信网络，需要继续补齐全局 API/WS token、白名单、反向代理鉴权或角色权限模型。

## 执行限制

### 单客户端互斥

默认情况下，同一个 Unity 客户端同一时间只允许一个执行上下文占用：

- server 会用控制锁阻止其他控制端抢占同一 Unity 客户端。
- Unity 本地上报 `clientBusy` 时，server 会拒绝新的普通单次执行。
- 请求序列执行期间，锁会保留到序列结束、失败、取消或超时。
- 控制端断线不会自动取消已提交执行；锁保留到执行结束或超时。
- 如果 Unity 注册的方法元数据包含 `allowParallelExecution: true`，或执行请求显式传入 `allowParallelExecution: true`，server 会把该执行视为只读并行请求，放宽同一客户端的锁、running 和本地 busy 拦截。该能力只应用于不改变 Unity 状态的查询类方法。

### 请求序列限制

| 限制项 | 当前值 / 行为 |
| --- | --- |
| `stepDelayMs` | 范围 `0` 到 `300000` ms，超出范围会被归一化。 |
| `stopOnFailure` | 默认 `true`。失败后停止后续步骤。 |
| step 数量 | 当前没有硬性上限，但至少需要 1 个有效 step。 |
| step 参数类型 | Web/qamcp 侧主要传字符串、数字和布尔值；server 最终归一化为字符串数组下发 Unity。 |
| step 超时 | `step.timeoutMs` 优先，其次序列顶层 `timeoutMs`，最后 `EXECUTION_TIMEOUT_MS`。 |

### 历史记录限制

server 内存中最多保留最近 `200` 条执行历史，用于 `/api/results`、Web 控制台和 qamcp 最近结果查询。服务重启后这部分内存历史不会保留，也不会从日志文件回放恢复。

长期追溯可以使用 `QA_LOG_DIR` 输出的 JSON line 日志文件；如果需要面向多用户、多 client 的长期查询和审计，可开启可选 MariaDB/MySQL 执行历史存档。该存档是旁路写入，不替换 `/api/results` 的内存视图。

```env
QA_EXECUTION_ARCHIVE_MYSQL_ENABLED=false
QA_EXECUTION_ARCHIVE_MYSQL_URL=
QA_EXECUTION_ARCHIVE_MYSQL_MAX_CONNECTIONS=5
QA_EXECUTION_ARCHIVE_QUEUE_SIZE=10000
```

- 默认关闭时不连接数据库。
- 开启但连接串为空或数据库不可用时，server 会记录错误并继续按原逻辑启动。
- 运行中写入失败只记录错误日志，不影响执行请求和结果广播。
- 测试环境和线上环境部署步骤见 [MariaDB 执行历史存档-部署与验证手册](./MariaDB执行历史存档-部署与验证手册.md)；完整设计见 [MariaDB/MySQL 执行历史存档-设计方案](./MariaDBMySQL执行历史存档-设计方案.md)。

### 停止限制

`stop_execution` 和 `stop_sequence` 当前只在 server 侧把 pending 执行或序列标记为取消，暂不能强制中断 Unity 内部已经开始执行的方法。Unity 迟到返回会按 late result 处理，不覆盖 server 已经收敛的最终状态。
