# 发布检查清单：构建验证与冒烟

本文档用于单仓发布前检查。发布负责人需要先确认根仓库改动已处理，再更新 `docs/发布版本矩阵-仓库Commit与协议版本.md`。

## 发布前检查

- [ ] 确认根仓库没有未处理的变更，或所有待发布变更都已纳入本次提交。
- [ ] 确认 `docs/发布版本矩阵-仓库Commit与协议版本.md` 已记录本次发布使用的仓库 commit。
- [ ] 确认本次是否有协议变更；如有，已按版本矩阵中的协议变更格式记录影响范围。
- [ ] 确认 Rust server 是本次验证使用的 register server。
- [ ] 确认 Node legacy 不参与本次发布验证；如需紧急回退，已参考 `docs/NodeLegacy服务端-冻结规则与回退指南.md`。

## 构建和检查命令

### registerserver

```powershell
cd registerserver/rustserver
cargo fmt --check
cargo check
cargo test
```

```powershell
cd registerserver/client
npm install
npm run build
```

Node legacy 已冻结，仅在需要紧急回退时检查：

```powershell
cd registerserver/server
npm install
node --check src/index.js
```

### qamcp

```powershell
cd qamcp
cargo fmt --check
cargo check
cargo test
powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\build-release.ps1
```

发布给 MCP 客户端时，优先分发：

```text
qamcp/dist/qamcp.exe
```

### Unity package

- [ ] 用 Unity 2022.3 打开 `unityclient/My project`。
- [ ] 进入 Play Mode，确认 Editor 默认启用 QA 功能。
- [ ] 确认 Unity 客户端自动注册到 register server。
- [ ] 构建或启动 Player 测试包时确认默认不连接；传入 `--qa-test-enable` 后再注册在线。
- [ ] 确认 `register`/`heartbeat` 包含 busy 状态字段。
- [ ] 执行一个静态 `[QaTest]` 方法。
- [ ] 执行一个 MonoBehaviour 实例 `[QaTest]` 方法。
- [ ] 确认普通方法本地 busy 时会拒绝第二个请求。
- [ ] 如本次涉及并行查询能力，确认 `AllowParallelExecution` 方法可以并行执行且不会修改 Unity 状态。

## 端到端冒烟验证

- [ ] 启动 `registerserver/rustserver`。
- [ ] 启动或构建 Web 控制台。
- [ ] Unity 进入 Play Mode 并注册在线；Player 冒烟时显式使用 `--qa-test-enable`。
- [ ] 使用 qamcp 调用 `qa_health`。
- [ ] 使用 qamcp 调用 `qa_list_unity_clients`，确认至少一个客户端 `available=true`。
- [ ] 使用 qamcp 调用 `qa_find_methods`，确认方法列表符合预期。
- [ ] 使用 qamcp 调用 `qa_get_method`，确认短 methodId 或显示名能解析到方法详情。
- [ ] 使用 qamcp 执行一个单步方法并收到 `qa_result`。
- [ ] 使用 qamcp 执行一个请求序列并收到 `sequence_finished`。
- [ ] 使用 Web 控制台执行一次方法，确认状态和历史记录正确。

## 发布产物

- [ ] `registerserver/rustserver` 可执行文件或部署包。
- [ ] `registerserver/client` 构建产物。
- [ ] `qamcp/dist/qamcp.exe`。
- [ ] Unity package 目录或 package 发布包。
- [ ] 更新后的根目录文档。

## 发布后记录

- [ ] 在 `docs/发布版本矩阵-仓库Commit与协议版本.md` 中补齐实际 commit。
- [ ] 在 `docs/变更记录-项目治理与文档调整.md` 中记录发布版本、发布日期、主要变更和已知问题。
- [ ] 如有远端仓库，运行 `.\push-all.ps1` 或 `git push` 推送当前仓库。
- [ ] 如有 tag，确保 tag 指向本次发布 commit。
